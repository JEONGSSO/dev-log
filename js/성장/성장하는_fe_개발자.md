# 장현석님

성장의 욕망을 통해 방향을 찾아야 한다

잘 하고 있는건지? 생각이 든다

### 성장을 해야한다고 말은 하지만 이유를 찾아본적 있나?

1. 솔직한 욕망 드러내기
   - 성장은 수단인가? 목표인가?
2. 목표에 맞는 방향 조준
   - 방향이 올바르다면 중간은 간다.
3. 직접 실행
   - 실행하지 않으면 아무것도 일어나지 않음.
4. 성장 확인
   - 적절한 피드백 & 피드 포워드

### 고민 위임

1. 생각할 수 있는 기회
   - 생산적인 생각을 할 수 있는 기회
2. 의미 있는 고민
3. 더 깊이 고민하기
4. 고민보다 실행

### 유효하지 않는 질문

<b>질문을 어떻게 하느냐가 제일 중요</b>

1. 스스로에게 질문
2. 정해진 답 확인하지 않기
3. 결정권 위임 하지 않기
4. 옳은 결정이 되도록 노력

### 모범 사례

1. 남과 비교하지 않기
2. 지름길은 X
3. 길을 잃고 헤메는 것(삽질)도 다 경험
4. 잘 하고 싶은 것이 있다면 그것을 하자

프론트엔드 개발자는 사용자의 환경과 경험을 개선해주는게 가장 큰 역할!

너무 기술적인 부분에만 매몰되어 있는게 아닌지 생각해보기.

# 김민준(벨로퍼트)님

내가 하고있는 분야에서 하나 하나 자세히 알아가보기

내가 하고있는 업무와 관련된 부분들이나 생산성을 올려주는 부분만 학습하고

그 외에는 여유롭게 학습하기

### 하나씩 채워가다 보면 뭘 해야 하지 하는 시점에 도달했을때

- 다른 개발자들과 소통
- 기술 컨퍼런스 참여로 인사이트 얻기
- 동영상과 해외자료 확인
- 꼭 새로운 기술 배우는것만이 성장은 아니다

### 내가 옛날에 썼던 코드 리팩토링 해보기

- 어떻게 하면 효율적으로 일할 수 있을까?
  - 공통된 로직을 추상화하여 라이브러리로 사용

### 비효율을 찾아내고 변경해보기

- 자동화
- 추상화

### 기획과 UI/UX 백엔드를 알아두면 정말 좋다

- 그들의 시각에서 바라 볼 수 있기때문
- 기본기 정도만 학습
- 어떤 도구를 사용해서 작업하는지?
- 협업 도구 피그마 추천 (사용법 리디 블로그에 올릴 예정)

### 사이드 프로젝트

<strong>하루에 일정 시간을 할당하여 꾸준히</strong>

개발 기간을 정하고 필수기능 먼저 도입 v1 -> v2 ....

- 성공적인 프로덕트? 익숙한 언어나 프레임워크 사용하기
- 새로운 기술? 트렌디 한것들 사용하여 학습 (규모는 작게)
- 실용성 있는 프로덕트 만들기(재미도 좋지만)

### 기획 프로세스 빠뜨리지 말기

- Low fi Prototype 종이에 끄적이는 정도의 프로토타입
- High fi Prototype 디자이너가 넘겨준 시안
- Low 여러개 -> High 변환

### 릴리즈 후 회고

- 어떤것을 했고
- 무엇이 아쉬웠고
- 다음번에 어떻게 개선할지 등등의 정보 기록

### 정리

1. 자신이 모르는것을 리스트업해보고 하나씩 풀어나가기
2. 내가 모르는게 무엇인지 알기위해 개발자들과 소통
3. 새로운 기술도 좋지만 지금 가지고있는 기술을 갈고 닦자
4. 백엔드와 디자이너 이해하기
5. 사이드 프로젝트 진행하기

# 변정훈(아웃사이더)님

## 오픈소스로 성장하는 개발자

### 매우 변화가 빠른 프론트 생태계 변화가 생겨도 괜찮으려면?

- 내가 사용하는 프레임워크나 라이브러리의 내부 구현을 알아두기

### 내부 코드를 보는 연습은 시간이 많이 걸린다

- 익숙해 질 수록 빨라진다
- (대부분은) 좋은 코드에서 보고 배울 수 있다
- 지금 커다란 프로젝트들의 초기버전을 보는것도 좋다 (초기 설계 배경을 볼 수 있기때문에)

### 결국 브라우저 위에 동작

- HTML
- 브라우저 렌더링
- JavaScript 엔진 (v8, SpiderMonkey)
- ECMAscript
- css 스펙

기술이 바뀌어도

- 비슷한 접근 방법을 더 개선하거나(react와 angluar를 보고 영감을 받은 vue)
- 같은 문제에 다른 해법 제시 (리액트는 단방향, 앵귤러는 양방향 등...)

점점 더 프론트 할 수 있는게 많아지고있다

- SPA
- JamStack
- Site Generator
- Headless CMS

### 서버와 별도로 배포

- 배포 파이프라인 구성
- 배포 구성 결정(서버나 스토리지, CDN)
- 브랜치/PR 별 프리뷰
